# 设计:skills-stats-page

## 锁定口径(需求访谈逐项拍板,实现时不得擅改)
- **主表只排 used**:equipped(装备态)不进总览页任何聚合;只在单 skill 详情页展示,
  并标注"仅 OpenClaw,带了≠用了";任何位置不得与 used 相加(沿袭既有约束)。
- **漏斗范围** = catalog 中 `type ∈ {own, meta}`(2026-06 时点共 21 个);
  external 与非公司库 skill 不进漏斗。
- **主表全量**:所有被使用过的 skill 都上榜(含非公司库),加"来源"列,
  取值 own / meta / external / 非公司库(catalog 查无此名)。
- **柱状图**:按 UTC 日,柱内按 skill 分段堆叠;取窗口内使用量前 8 的 skill 各占一色,
  其余合并为灰色"其它"段;悬浮图例高亮该 skill 全部分段,悬浮柱子出当日明细。
- **时间窗筛选只控制柱状图**(默认 30 天,可选 7 / 90 / 全部);
  主表固定 7 天/30 天/累计三列;漏斗第 3 层固定 30 天。
- **下钻** = 独立视图 + 返回,与现有 agent 详情页交互一致。
- **无概览指标卡**(访谈中明确删除,页面直接从柱状图开始)。

## 改动文件与职责

### `server/app.py`
1. **catalog 同步**:后台定时拉取(默认每小时,地址默认
   `https://github.com/tranfu-labs/tranfu-skills/releases/download/catalog/index.json`,
   可由环境变量覆盖以便测试)。成功 → 更新缓存(内存 + SQLite 单行表,记 `fetched_at`);
   失败 → 沿用旧缓存并对外暴露过期标记。启动时异步拉一次,不阻塞启动。
2. **`GET /api/skills?days=30`**(总览,days ∈ {7,30,90,0=全部} 仅影响 daily):
   - `daily[]`:窗口内逐 (日 × skill × runtime) 的 used 会话数。原样返回明细聚合,
     Top8+其它 的合并和筛选过滤都在前端做(量级:90 天 × 几十 skill × 4 runtime,很小)。
   - `table[]`:每 skill 一行(只含 used):名称、来源(catalog 映射)、7 天/30 天/累计会话数、
     30 天用户数(去重 operator)、各 runtime 会话数、近 14 天逐日序列(画趋势小图)、最近使用日。
     默认按 30 天降序,平手按累计。
   - `funnel`:own+meta 的三层**名单**(不只是数字):catalog 收录 / 已安装 / 30 天有人使用,
     外加服务端直接算好的闲置名单(已安装 − 30 天使用),并带 catalog 的 `fetched_at` 与过期标记。
3. **`GET /api/skill/{name}`**(详情):指标(7/30/累计/用户/首见/最近)、
   日级序列(used 与 equipped 分列)、runtime 分布、operator 分布、
   最近记录(默认 50 条:日期/operator/runtime/模式/session_id)、来源。查无此名 → 404。
4. **安装态数据源**:复用 profiles 表的 skills 字段(各 agent 最新配置快照),
   与 catalog 按名字匹配。

### `frontend/`
- 导航:顶部加第三个 SKILLS 标签;路由扩展出 `/skills` 与 `/skill/:name`。
- skills 总览视图:进入时拉 `/api/skills`,之后约 10 秒刷新(不进主轮询);
  筛选条、SVG 手绘堆叠柱状图(零依赖,不引入图表库)、可排序主表、漏斗三层条 + 展开名单;
  空库显示空态,接口失败显示错误态,不报错不留洞。
- skill-detail 视图:拉 `/api/skill/{name}`;used 柱 + equipped 折线同图但不堆叠;
  分布两栏;最近记录表;返回按钮回总览。
- 看板:删除右侧栏 skills 区块及其渲染调用(`/api/state.skills` 字段保留,前端不再消费)。
- 暗/亮双主题与 ≤600px 窄屏遵循现有样式约定。

### 文档
- PROTOCOL.md 如记录读侧接口则补两个新端点;上线后 spec delta 合入 `openspec/specs/board/spec.md`。

## 性能与降级
- 聚合均读时现算(与 `skill_usage()` 现风格一致);`skill_uses` 年量级数万行,按日分组足够快;
  新端点不进 2 秒轮询,无缓存层(过早复杂化,单容器约束)。
- catalog 拉取失败:漏斗用旧缓存渲染 + 过期标记;从未成功过 → 漏斗区显示"公司库目录不可达",
  页面其余区块不受影响。

## 实现前必须先验证的两件事
1. profiles 表 skills 字段里的名字与 catalog 的 `name` 是否同一套写法(漏斗第 2 层靠名字匹配);
   抽样比对,不一致则先定归一化规则。
2. profiles 安装态是"最新快照"而非历史:漏斗反映当前安装情况,界面注明此口径。

## 依赖与顺序
- 数据已就绪(track-skill-usage 与三条 runtime 链路均已合入),无前置变更。
- 实施顺序:服务端总览接口 → 总览视图 → 详情接口+视图 → 最后移除看板侧栏
  (避免中间态出现"哪里都看不到排行"的窗口期)。
