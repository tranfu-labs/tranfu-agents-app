# 提案：fix-agent-active-duration-overcount

## 背景

线上 Agents 列表出现单 Agent 单日运行时长超过 24 小时、当天累计接近 20 小时但缺少相应连续在线证据等异常。排查确认问题不是前端格式化，而是活跃时长聚合直接相加同一 Agent 的重叠 session，并把超过 180 秒的心跳断档继续累计到后续事件；同状态、同步骤在长断档后恢复时，写侧还会覆盖原事件的 `last_seen`，永久丢失断档边界。

明细最右列展示的是“当前时间距上次活跃”，现有“最近活跃 / Last active”文案也容易被误解为另一段运行时长。

## 提案

- 以最终 Agent 身份 `operator + agent||runtime` 为单位，对全部 session 的连续活跃区间按上海统计日切分后取并集；排行、趋势、八卡、Agents 明细和 Agent 详情复用该结果。
- 以 `STALE_SECONDS=180` 为连续性边界：事件与上一条已确认心跳断档超过 180 秒时，旧段停在最后确认心跳，新存活事件从自身服务端接收时间开始新段。
- 同状态、同步骤在长断档后恢复时插入新事件行，保留断档边界；正常高频心跳继续批量更新 `last_seen`，不进入活动流。
- 迟到终态只关闭当前连续段，不回填断线区间。
- 将 Agents 明细列文案明确为“距上次活跃 / Time since last active”。
- 不执行历史数据改写；能由现有原始事件识别出的重叠与断档在读侧自动恢复。

## 影响

- 服务端 board 域：活跃区间构造、身份级区间并集、上海统计日切分和所有既有时长派生字段。
- 服务端 ingest 域：纯心跳去重增加长断档恢复例外。
- 前端 Agents 明细：中英文列标题与手机 `data-label`。
- OpenSpec、ADR-0003/0013、模块地图、项目操作手册和 Agents 线框事实源。
- SQLite schema、事件协议字段、shim 和历史数据均不变。
