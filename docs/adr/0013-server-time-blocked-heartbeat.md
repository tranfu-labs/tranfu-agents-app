# ADR-0013 活跃时长用服务端时间;blocked 计活跃且单列;心跳 60s / stale 180s

- 状态:Accepted
- 关联:PROTOCOL.md §1 §6、ADR-0003(心跳去重)

## 背景 / 问题
1. 活跃时长原先用客户端 `ts` 做区间累加。多机时钟漂移会污染指标(虚高/虚低),`b<=a`
   只挡负值挡不了偏移。
2. `blocked` 在枚举里有定义,但既不计活跃也不算终态 —— 卡住的 agent 在统计里凭空消失。
3. 心跳周期未定义,而服务端 stale 阈值硬编码,wrapper 不知道该多久发一次,易被误判 idle。

## 决策
- **时间口径:** 事件落库时记服务端权威时间 `recv`;所有时长/活跃度计算一律用 `recv`,
  客户端 `ts` 仅供展示。客户端无需 NTP。
- **blocked 归属:** 视为**存活态** —— 计入活跃时间、不翻 idle(它确实占用一个 run);
  同时质量块**单列** `blocked` 计数供治理看"谁卡住了"。即"既计活跃、又单列"。
- **心跳:** 推荐存活态每 **60s** 发一次心跳;服务端 stale 阈值 = **180s**(3 个周期)。
- **连续段:** 相邻事件距最后确认心跳超过 180s 时,旧段停在最后确认心跳;恢复的存活事件从自身
  `recv` 开新段,迟到终态不得回填断线期间。
- **身份墙钟时长:** 同一最终身份 `(operator, agent||runtime)` 的多个 session 区间先取并集,
  再按 `Asia/Shanghai` 日切分;不得直接相加重叠 session。

## 后果
- ✅ 跨机部署的活跃指标稳定可信。
- ✅ blocked 不再丢失,且治理可见。
- ✅ 单 Agent 单统计日时长自然不超过 24 小时,并行 session 不会重复计时。
- 约束:`ACTIVE_ST = running/started/waiting/blocked`,改判活逻辑须同步此集合。
