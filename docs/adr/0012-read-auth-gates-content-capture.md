# ADR-0012 读侧鉴权是内容上报的硬前提(服务端强制)

- 状态:Accepted
- 关联:add-read-access-control(读侧鉴权)、PROTOCOL.md §5、ADR-0002

## 背景 / 问题
看板默认"谁有网址谁就能看"。一旦开启敏感上报(`input`/`output`/`instructions`/`memory`),
等于把 prompt、代码、系统提示明文挂到对外可见的页面。仅靠"建议"配置鉴权不可靠。

## 决策
把"开启内容捕获前必须先有读侧鉴权"从建议**升级为硬约束,并由服务端强制执行**:

- 服务端通过 `TF_READ_KEY`(应用内只读令牌)或 `TF_READ_AUTH=1`(声明边缘
  Cloudflare Access / Caddy 已就位)判断读侧是否受保护。
- 若读侧**未**受保护,收到含敏感字段(`SENSITIVE_KEYS`)的事件时,服务端**丢弃这些字段不予存储**,
  只保留状态类字段。即便有人误开捕获,也不会造成明文裸奔。

## 后果
- ✅ 默认安全:不配鉴权就拿不到敏感内容,无侥幸空间。
- 业务流程影响(知情):内容捕获从"改环境变量就开"变为"先配读侧鉴权、再开捕获"两步;
  新人入职默认全关,开捕获需先完成部署侧鉴权并走团队审批。摩擦是刻意的。
- 约束:禁止以任何方式绕过该 gate 存储敏感字段。
