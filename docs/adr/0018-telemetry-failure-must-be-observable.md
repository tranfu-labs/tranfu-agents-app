# ADR-0018 遥测失败必须可观测,绝不静默冒充成功

- 状态:Accepted
## 背景
最坑的一次:`tf_report.py` 在 `TF_SERVER` 为空时走的是 `--print`(dry-run)分支——**只打印 payload、不 POST、`exit=0`、不进 spool**。从外面看是"成功了",实际一条都没发。叠加 hook 跑非交互 shell 读不到身份(ADR-0016),造成"hook 装好了、看板却空"的假象,并导致多轮**误判"已上报"**、反复说"重启就好"。
## 决策
- 遥测的"**没配好 / 没发出去**"必须有**可观测信号**:非零退出、错误日志、或显式落 spool(`~/.tranfu/spool.ndjson`)。**不得**用"`TF_SERVER` 为空 → 静默打印 → exit 0"这种分支冒充成功。
- "best-effort、不阻塞 host agent"仍是底线(遥测绝不能让 agent 崩),但**失败要留可查证据**,而不是无声 return。
- **部署 ≠ 合并**:代码进 main 不等于线上生效。部署后用**版本探针**确认服务端分发的 shim 是新版(例如 `curl $SERVER/shims/tf_hook.py | grep <新符号>`),再宣布"已上线"。
## 后果
- ✅ 失败立刻暴露,诊断靠探针而非猜测。
- ✅ "代码在 main"与"server 已部署"被当两件事对待,避免"以为部署了"。
- 约束:任何静默 `return`/`pass` 前,必须确认它不是把"没发出去"当成"发成功了";新增的上报路径都要有失败可见性。
